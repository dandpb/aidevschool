# ADR: Spaced-Repetition + Streak Layer for codexDojo / pixelDojo

- **Status:** Proposed
- **Date:** 2026-06-21
- **Supersedes:** the hardcoded `_next_reviews()` stub in `learner/substrate/dashboard_snapshot.py`
- **Research basis:** `/deep-research` run `wf_f154f0ca-00a` (Duolingo mechanics → programming-mastery game). 19/25 claims adversarially verified.

---

## TL;DR

Replace the synthesized `_next_reviews()` stub with a real **FSRS** (Free Spaced Repetition Scheduler) layer in the Python substrate, and formalize the half-built `retry_count`/`learning_states` fields into a **streak + freeze** layer. Both live in `learner/substrate/`; codexDojo and pixelDojo only **render** the derived views — they never schedule or decide mastery. This matches the existing rule that the filesystem is the source of truth and the producer never verifies itself.

The one constraint that anchors everything: **streaks reset only on a missed executable-evidence gate, never on a calendar lapse**, and arbitrary scarcity (hearts/lives/leagues) is explicitly excluded — the verified research shows those mechanics harm coding-task motivation.

---

## Context

The `AI Code Dojo` vision (`docs/PROMPTS/00_IDEIAS_opencode_gpt55.md`) wants a Duolingo-style loop for programming mastery. The deep-research run isolated **which Duolingo mechanics survive verification** and which don't:

### Confirmed as retention drivers (build these)
1. **Spaced repetition** — Duolingo's HLR scheduler gave **+12.0% (p<0.001)** daily retention over 3.3M students (Settles & Meeder, ACL 2016). The spacing effect is among the most robustly replicated findings in cognitive psychology, and extends (medium confidence) to higher-order reasoning — i.e. it can plausibly build durable *code-review/architecture judgement*, not just syntax recall.
2. **Streaks** — most powerful mechanic; early streaks motivate via incremental achievement, long streaks via loss aversion.
3. **Streak Freeze (slack)** — *increases* daily activity (+0.38% at 1→2 freezes); caps at 2 (diminishing returns).
4. **CURR (Current User Retention Rate)** — the Markov transition metric that drove 4.5× DAU growth. The repo *already* has its analog: the 4-state `learning_states` machine (presenting / practicing / evaluating / mastered).

### Refuted or harmful (do NOT build)
- **Hearts / lives** — no strategic analog in code evaluation → "tacked-on nuisance" (Duolingo's own failed "moves counter" was "completely neutral"). The gate is the only legitimate scarcity.
- **Leagues / leaderboards** — the specific league numbers failed verification (1-2); the "dark side of gamification" literature lists leaderboards as a top cause of *performance loss*.
- **R-value course tuning (R=√(S/L))** — the Hakulinen-style analysis failed 0-3. Drop the entire framework.

### Existing substrate already has the seeds
- `state_machine.learning_states` (presenting/practicing/evaluating/mastered) → **is** the CURR basis.
- `active_unit.retry_count` / `retry_limit` → streak/freeze seed.
- `units_log: []` → the FSRS review-history feed, currently empty.
- `LearnerSnapshot.nextReviews` (`domain.ts`) → the render surface, currently fed by a **hardcoded stub**.
- `empirical_gates` (coverage ≥80%, mutation ≥60%, benchmark CV<20%) → the gate that streaks reset on.

So this is not greenfield: it formalizes a stubbed surface into a real layer.

## Decision

### 1. FSRS lives in the substrate, not the app

Use **`py-fsrs`** (Python; matches `learner/substrate/`). The TS apps (`codexDojo`, `pixelDojo`) **never** schedule reviews — they consume derived views generated by `dashboard_snapshot.py`. This preserves:
- Rule #4 (filesystem is source of truth).
- The producer-≠-verifier rule (the agent that produces evidence is not the agent that schedules re-exposure).
- One scheduler, one review log, consumed by every engine.

`ts-fsrs` is **rejected** for scheduling to avoid duplicating the algorithm across languages. (If a future pixelDojo arcade game needs client-side scheduling for offline play, `ts-fsrs` may be used as a *read-only projection* of the same `units_log` — same parameters, never the source.)

### 2. The review history feed: `units_log`

Promote `units_log` from an empty list to the FSRS input. Each entry is a **review record** for one unit (a learnable atom: a concept, a smell, an architecture pattern):

```yaml
# learner/learning_state.yaml — proposed extension
units_log:
  - unit_id: U0-sonda-rate-limiter-robustness
    concept: "token-bucket robustness"
    kind: concept            # concept | smell | architecture | pitfall
    reviews:
      - { date: 2026-06-15, rating: again, elapsed_days: 0, state: presenting }
      - { date: 2026-06-17, rating: good,  elapsed_days: 2, state: practicing }
      - { date: 2026-06-21, rating: easy,  elapsed_days: 4, state: evaluating }
    fsrs:                     # per-card scheduler state (py-fsrs Card fields)
      stability: 4.7
      difficulty: 0.22
      due: 2026-06-28
      last_review: 2026-06-21
    mastered: false
```

**The `rating` only ever comes from the executable gate, not self-report.** A unit earns `again`/`good`/`easy` by how it performs against `empirical_gates`, not by how the learner *feels*. This is the single most important rule — the research found learners systematically misjudge spaced practice as ineffective, so subjective ratings would corrupt the scheduler.

| Gate outcome | FSRS rating |
|---|---|
| Gate fails (coverage/mutation/benchmark not met) | `again` |
| Gate passes, retry_count > 0 on the way | `hard` |
| Gate passes first try | `good` |
| Gate passes + exceeds target (e.g. mutation ≥80%) | `easy` |

### 3. Streak + freeze layer

Add to `learner/learning_state.yaml`:

```yaml
streak:
  current: 12
  longest: 18
  last_gate_date: 2026-06-21     # the date a gate was last satisfied
  freezes:
    equipped: 2                  # hard cap 2 (research: 3 ≈ 2, diminishing returns)
    max: 2
```

**Reset semantics (the decision with the most evidence behind it):**
- The streak increments when a unit **passes its executable gate** (`last_gate_date` moves to today).
- The streak **breaks** only when a day ends with `last_gate_date` older than yesterday **and** no freeze available. A missed day with a freeze available consumes one freeze instead.
- Streaks do **not** reset on calendar idleness alone, nor on a gate *attempt* that fails — only on an unabsorbed missed gate-day. Early streaks (<10) get the most motivational leverage from the "incremental achievement" framing, so the UI should celebrate small early gains.

This makes the **gate the scarcity**, not an artificial lives counter — directly implementing the verified "don't port hearts" finding.

### 4. CURR-style metric for the loop

Reuse the existing `learning_states` machine as the Markov basis. Define a derived **CURR** for the programming learner:

> **CURR = share of units in `practicing` or `evaluating` state that advance (or sustain) within the rolling 7-day window**, i.e. that either pass a gate or remain actively scheduled rather than going stale.

This is computed in the substrate (it already aggregates from `units_log` and `BACKLOG_STATUS.md`) and surfaced as a single number on the dashboard. Treat the exact "current" definition as **unvalidated** (flagged open question in the research) — instrument it, log it, and revisit once real data exists. Do not over-index on the number.

### 5. Render surface (codexDojo `domain.ts`)

`LearnerSnapshot.nextReviews` already exists; extend the reason vocabulary and add streak:

```ts
// engines/codexDojo/src/domain.ts — proposed additions
readonly nextReviews: ReadonlyArray<{
  readonly unitId: string
  readonly title: string
  readonly dueIn: string
  readonly reason: "overdue" | "due" | "interleaving" | "recurring-trap"  // was: overdue|interleaving|recurring-trap
}>
readonly streak: {
  readonly current: number
  readonly longest: number
  readonly freezesEquipped: number
  readonly freezesMax: number
  readonly daysToBreak: number | null   // null when a freeze absorbs the next miss
}
readonly curr: number                   // 0..1, the CURR-style transition metric
```

`dashboard_snapshot.py` populates these from `units_log` (via `py-fsrs`) and `streak`. The hardcoded `_next_reviews()` is **deleted**.

pixelDojo reads the same derived snapshot (or a slice of it) so arcade games surface the same `nextReviews` / streak — one loop, many surfaces.

## Consequences

**Positive**
- A real scheduler replaces a stub; reviews become evidence-driven, not invented.
- Streak/freeze semantics are anchored to the executable gate, not arbitrary timers.
- One scheduler in one language; both engines render the same truth.
- Directly implements the four verified retention levers and excludes the three refuted ones.

**Negative / trade-offs**
- New runtime dependency: `py-fsrs`. Must be added to the substrate's environment and `python3 -m learner.substrate` must still be the regeneration command.
- `units_log` now carries real per-card scheduler state → schema migration (the empty `[]` today means there's no data to migrate, but `schema.yaml` invariants must extend).
- The CURR definition is an unvalidated proxy. It is explicitly labeled as such — do not let it drive automated decisions until validated.

**Risks explicitly declined**
- No hearts/lives, no leagues, no gems, no R-value course tuning. Each was either refuted by the research or shown to harm motivation for non-strategic tasks. The gate remains the only scarcity.

## Schema deltas (acceptance contracts)

### `learner/substrate/schema.yaml`
- Add `Streak` type with invariant `0 <= streak.freezes.equipped <= streak.freezes.max <= 2`.
- Add invariant: a `review.rating` must be derivable from `empirical_gates` outcomes (i.e. the producer of the rating is the gate, not the learner).
- Extend `LearningState` invariants: a unit with `fsrs.due <= today` and no passing review since is `practicing` or `evaluating` (eligible for re-exposure), never silently `mastered`.

### `learner/learning_state.yaml`
- Populate `units_log` with real review records (FSRS cards).
- Add the `streak:` block.
- Add `curr:` (derived; may live in the generated snapshot only rather than hand-edited YAML — prefer computed).

### `engines/codexDojo/src/domain.ts` + `dashboard_snapshot.py`
- Extend `LearnerSnapshot` as above; update **both** the TS type and the Python renderer together (existing contract: shape locked at `domain.ts:LearnerSnapshot`).

## Rollout (incremental — matches the engine's "start small" rule)

1. **Phase 0 — Log, don't schedule.** Populate `units_log` review records from existing gate outcomes (the agents already evaluate). No FSRS yet; `due` is a naive +N days. Delete the hardcoded `_next_reviews()`.
2. **Phase 1 — Introduce `py-fsrs`.** Seed scheduler state from the Phase-0 log; `due` becomes FSRS-computed. Surfaced on the codexDojo dashboard only.
3. **Phase 2 — Streak + freeze.** Wire the gate outcomes to streak increments/breaks; render streak on dashboard.
4. **Phase 3 — pixelDojo surface.** Expose the same `nextReviews`/streak slice to arcade games so a quest can *be* a scheduled review.
5. **Phase 4 — CURR.** Compute and display the metric; label it unvalidated; revisit after real data.

## Acceptance criteria (verification, not "looks right")

- [ ] `python3 -m learner.substrate` regenerates `engines/codexDojo/src/data/learner.ts` with FSRS-derived `nextReviews` (no hardcoded values remain).
- [ ] A unit whose gate fails produces a review with `rating: again` and an FSRS `due` ≤ tomorrow; a unit that passes with high margin produces `rating: easy` and a longer interval. Asserted in `learner/substrate/tests/`.
- [ ] `streak.freezes.equipped <= 2` is enforced as a schema invariant and a missed gate-day with a freeze available consumes exactly one freeze without breaking the streak.
- [ ] `pnpm run lint && pnpm run test && pnpm run build` pass in codexDojo after the `domain.ts` extension.
- [ ] No code path introduces hearts, lives, leagues, or gems. (Negative acceptance criterion — grep the diff for these terms and confirm absence.)
- [ ] The CURR number is rendered with an explicit "unvalidated proxy" label.

## Open questions (deferred to data, not to this ADR)

1. Does spacing actually build durable *architecture judgement*, or only recall? Instrument on real learners before claiming it.
2. What is the right definition of "current" for a programming learner (daily commit vs weekly passed-test)? Phase 4 revisits once logged.
3. Is there a coding analog of the strategic-decision tension (which hearts fail to provide)? Candidate: timed constraint-tradeoff architecture choices — validated separately, not assumed.
