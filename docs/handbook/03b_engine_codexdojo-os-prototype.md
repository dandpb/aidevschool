# Engine ‚Äî codexDojo OS

`engines/codexdojo-os-prototype/` is a browser-based educational Linux desktop.
It explores how operating a desktop, terminal, file browser, architecture map,
and app catalog can teach computing fundamentals in context.

This package owns the educational OS bounded context. It complements the `codexDojo` dashboard and
receives the same canonical learner snapshot through a separate generated, read-only module.

## Run the engine

The package uses npm, keeps its own lock file, and requires Node.js 20.19+ or
22.12+ because the lock currently resolves Vite 8.1.4:

```bash
cd engines/codexdojo-os-prototype
npm install
npm run dev
```

The OS dashboard starts on its normal Vite address. The Engine Hub uses fixed
development fallbacks for the embedded dashboard (`5175`), PixelQuest (`5176`),
and all 16 voxelDojo game packages (`5177`, `5202`, `5203`, `5205`‚Äì`5209`, and
`5211`‚Äì`5218`).
Configure `VITE_CODEXDOJO_URL`, `VITE_PIXELDOJO_URL`, and the compatibility
`VITE_VOXELDOJO_URL` for HASH RING. Use `VITE_VOXELDOJO_URLS` as a JSON map from
game ID to URL when deploying the full catalog.

Open the address printed by Vite. Build the production bundle with:

```bash
npm run build
```

## Current scope

The current implementation includes:

- a desktop shell with a top panel, dock, searchable launcher, and movable
  windows;
- six available app surfaces: Dojo Tracks, Terminal, Files, Architecture Map,
  App Center, and Engine Hub;
- six external engine adapters: three embedded browser engines and three fixed,
  read-only local actions; voxelDojo includes all 16 game packages;
- a catalog of planned apps with explicit maturity states;
- a contextual Learn Mode and a deterministic local mentor interaction;
- responsive layouts for desktop and smaller screens.

The Vite development server and integrated local preview own the loopback action
bridge. It uses a same-origin session bootstrap, a per-process token, fixed
Python argv, JSON-only requests, and single-action concurrency. Normal static
production builds render local actions unavailable; `npm run
preview:integrated` builds and serves the local integrated form. The engine has
no deployed backend, persistent virtual filesystem, external mentor provider,
or app SDK yet.

## Learner-state boundary

The top bar and Dojo app render a generated learner snapshot from
`learner/learning_state.yaml`. The React runtime never writes canonical state. Local missions,
catalog state, terminal actions, and mentor responses remain UI demonstrations; they do not emit
verifier-approved evidence or mark a unit `mastered`.

PixelQuest and voxelDojo can forward their existing raw evidence record to the
embedding Hub. The Hub accepts only messages from the selected iframe and its
configured origin, preserves the verifier-required label, and doesn't write
learner state. Embedded game builds set `VITE_CODEXDOJO_OS_ORIGIN` to the exact
OS origin; emitters reject a mismatched embedding referrer.
