## 1. Go — `go-impl/`

### 1.1 Idioms used

- **`sync.Mutex` over `map[string]*ClientBucket` as the only state
  primitive.** The critical section is short (one map lookup + a few
  float ops) and the keyspace is unbounded, so `sync.Map`'s "read-
  heavy, stable keys" sweet spot doesn't apply.
- **Package-level `Clock` interface** (`interface { Now() time.Time }`)
  injected at construction. Default `RealClock{}`; tests use a
  `fakeClock` with an `Advance(d)` method. The interface has one
  method, which is the Go-idiomatic "small interface" pattern.
- **`slog` with `NewJSONHandler`** for structured logging. No
  global config — the `*slog.Logger` is threaded through
  `loggingHandler` and created in `main`.
- **Middleware as `func(http.Handler) http.Handler`**, composed via
  `mux.Handle("/", rl.Middleware(serveWelcome))`. The same pattern
  is used in `net/http` and in the wider Go ecosystem (chi, gin).
- **`signal.NotifyContext(ctx, syscall.SIGINT, syscall.SIGTERM)`** for
  shutdown signaling. `srv.Shutdown(shutdownCtx)` is the standard
  graceful-shutdown sequence.
- **`context.Background()` cancellation as a `defer stop()`** to
  release the signal handler when `main` returns.
- **Manual env parsing with `slog.Warn` fallback** — every misconfig
  logs and continues, so a misconfigured deployment still starts.

### 1.2 Strengths in this domain

- **The stdlib `net/http` server is production-grade out of the
  box.** `ReadHeaderTimeout`, `ReadTimeout`, `WriteTimeout`,
  `IdleTimeout`, and `Shutdown` are all there. No third-party
  framework needed.
- **The race detector is the killer feature.** `go test -race` will
  catch a missed mutex or a copied struct on the first run. No
  equivalent in Node or Rust (Rust's `loom` is opt-in; Node has
  no equivalent).
- **Static binary, tiny image (~13 MB).** The 2-stage Docker build
  is the simplest of the three.
- **Compilation is a soft specification.** Type mismatches in
  middleware composition fail at `go build`, not at request time.

### 1.3 Weaknesses in this domain

- **The error-type story is "values, not enums".** `error` is an
  interface; you can't exhaustively match on it. `errors.Is` and
  `errors.As` help but are not as ergonomic as Rust's `match` on
  `Result`.
- **No `try`/`catch`.** A panic in a handler crashes the goroutine
  (which is fine for `http.Server` because it recovers), but the
  developer ergonomics for "I expected this to fail" are clunky.
- **The `defer stop()` pattern in main is easy to forget.** A
  developer who copies the boilerplate without the `defer` will
  leave a leaked signal handler that prevents the process from
  re-binding to a port.
- **No "X-Forwarded-For" out of the box** — see
  `[GO-MAJOR-001]`. The Go stdlib gives you the raw socket, and
  trusting the header is a security decision the dev has to make.

### 1.4 When to pick Go for this kind of problem

- **Mid-size service that needs concurrency safety with minimum
  ceremony.** `go test -race` is the cheapest way to validate that.
- **Server that must start fast and stay small.** Cold start ~5 ms,
  image ~13 MB.
- **Team comfortable with "many small interfaces" and explicit
  error handling.** Go's idiom is *not* "easy"; it's *honest*.

## 2. Rust — `rust-impl/`

### 2.1 Idioms used

- **`std::sync::Mutex` inside `Arc<RateLimiter>`.** The type
  system enforces the share-but-don't-mutate pattern at compile
  time. `Arc` is `Send + Sync` iff the inner is, so the limiter is
  shareable across axum tasks automatically.
- **`Clock` trait with `Send + Sync` supertraits** so the limiter
  can be used from any thread. `MockClock` wraps a
  `Mutex<Instant>`, which is `Send + Sync` because `Instant` is.
- **`Decision` as a sum type** (`enum Decision { Allowed { … },
  Denied { … } }`). The middleware matches on the variant to
  decide which headers to stamp. Exhaustive match is enforced by
  the compiler.
- **`thiserror::Error` for the `AppError` enum** — every variant
  is `#[error("...")]`-formatted, so `eprintln!("{e}")` is
  self-describing.
- **The `start_instant` + `start_system_time` trick.** Captures
  both clocks at construction so `instant_to_system_time` is
  deterministic under a `MockClock` *and* correct under the
  system clock. See `[RUST-EDU-002]`.
- **`axum::Router` with `.route_layer(from_fn_with_state(...))`**
  to apply the rate-limit middleware only to the `/` route, not
  to `/status`. The order of `.route` and `.route_layer` matters.
- **A `tower::ServiceExt::oneshot` integration test harness**
  that drives the real axum router without binding a port. This
  is the canonical way to test axum apps in CI.
- **Tokio's `interval` for the cleanup task** with
  `tick.tick().await` to skip the immediate first tick.

### 2.2 Strengths in this domain

- **The type system is the killer feature.** A function that
  returns `Result<Decision, AppError>` cannot be called without
  handling both cases. The compiler enforces the 12-vocabulary
  contract of the rate limiter.
- **Zero-cost abstractions.** `Arc<Mutex<…>>` is two pointer
  indirections and an atomic — there is no runtime cost beyond
  what C would have. Compared to a Node `Map` (which is a
  hash table with V8 object headers) or a Go `map` (which is a
  hash table with runtime-managed buckets), Rust's
  `HashMap<IpAddr, ClientBucket>` is the smallest and fastest.
- **The `clippy --all-targets -- -D warnings` gate is the
  Rust equivalent of `go test -race`.** It catches a huge
  surface area of bugs (unused imports, `unwrap()` in the
  wrong place, inefficient patterns) without running the
  code.
- **`#[ignore]`d tests are visible in the test summary.** A
  reader of `cargo test` output sees `19 passed, 1 ignored`
  and is prompted to ask why.
- **The `Result`-as-a-sum-type pattern forces a thinking
  discipline**: every error variant must have a handling
  site. This is what makes Rust services "boring" in the good
  sense.

### 2.3 Weaknesses in this domain

- **Async test runtime fragility.** The `#[ignore]` on
  `concurrent_requests_never_overconsume` is a real cost — the
  test that would catch a race condition is the test that
  doesn't run. The `loom`-based alternative is opt-in and adds
  build time.
- **Compile times.** `cargo build --release` is minutes, not
  seconds. This is a productivity tax for small services.
- **The `Mutex::lock().expect("poisoned")` pattern silently
  crashes on poisoning.** In a long-running service, this is
  reasonable (poisoning means a previous holder panicked; the
  process is in an unknown state), but the dev who hits it
  has to know that.
- **The `arc_mutex` pattern is verbose for what it expresses.**
  A 5-line "increment a counter under a lock" in Go becomes 10
  lines in Rust. For a 200-line service the verbosity is
  acceptable; for a 2 000-line service it adds up.

### 2.4 When to pick Rust for this kind of problem

- **Long-running server where correctness and performance are
  non-negotiable.** The type system catches the bugs that
  stdlib-based languages discover in production.
- **Service that needs a small memory footprint and a small
  binary.** axum + tokio binaries are typically 5-15 MB.
- **Team that values "boring is good" and is willing to pay
  compile time for it.** Rust's value proposition is "if it
  compiles, it's probably correct." If you can't afford the
  compile time, you can't afford the value.

## 3. Node/TypeScript — `node-impl/`

### 3.1 Idioms used

- **Pure class as the core (`TokenBucketRateLimiter`)** with
  `#`-prefixed private fields. No Express, no I/O — the class
  is callable from any transport.
- **Function-type clock**: `type Clock = () => number`. The
  production clock is `Date.now`; tests use `vi.fn(() =>
  clockValue)` from vitest.
- **`express` for HTTP** with `app.disable('x-powered-by')` (no
  fingerprinting) and `trust proxy` set conditionally based on
  `TRUST_PROXY`.
- **Zod for config validation.** `RawEnvSchema.safeParse(env)`
  returns a `Result`; on failure, `ConfigError` is thrown with
  a `JSON.stringify(parsed.error.flatten().fieldErrors)`
  message so the operator sees all bad envs in one go.
- **`pino` for structured logging.** The level is set via
  `LOG_LEVEL`; the base is `{ service: 'rate-limiter-node' }`
  for log-aggregator routing.
- **`setInterval(...).unref()` for the cleanup timer** so the
  process can exit even if the timer is still scheduled.
- **Custom `RateLimiterError` hierarchy** with a stable
  `code` field for matching in logs and error reporters.
- **`supertest` for integration tests** — drives a real HTTP
  client against the in-process Express app, so the protocol
  layer is exercised end-to-end.

### 3.2 Strengths in this domain

- **The "pure core, thin shell" pattern is at its most natural
  in TypeScript.** A `class` with private fields and zero
  framework imports is the unit of testability. The integration
  test (`server.test.ts`) and the unit test
  (`rateLimiter.test.ts`) can be written and run independently.
- **Zod makes config validation a one-liner.** A schema is
  declarative; the error message is consistent; the
  `safeParse → flatten → ConfigError` pipeline is a teachable
  pattern.
- **V8 is fast.** For a single-threaded workload with mostly
  hash lookups and float math, the Node impl will be
  competitive with Go and Rust in raw throughput (the
  benchmarker will confirm).
- **`Object.setPrototypeOf(this, new.target.prototype)` in the
  custom error class** is the standard pattern for preserving
  `instanceof` after transpilation. The Node impl gets this
  right; many don't.
- **`app.disable('x-powered-by')`** — a small but principled
  defense against trivial fingerprinting.

### 3.3 Weaknesses in this domain

- **Single-threaded is also single-process.** `node:cluster`
  workers do not share state. A 4-worker deployment has 4
  independent maps, so a 10-token bucket actually has 40
  effective tokens. The README needs to make this explicit
  (see `[NODE-MAJOR-004]`).
- **The `Map` API is a 2-call "get or create".** `let b =
  map.get(k); if (!b) { b = ...; map.set(k, b); }` is two
  hash lookups. In Go and Rust the equivalent is one call
  (`or_insert_with` / `entry().or_insert_with()`). The
  performance difference is small but real.
- **No compile-time safety.** A typo in a property name is a
  `TypeError` at request time, not a build failure. The
  `tsconfig.json` with `"strict": true` mitigates this but
  doesn't eliminate it.
- **The `process.on('uncaughtException', err => process.exit(1))`
  pattern is the only safety net** for "something went wrong
  in async code I forgot to await". Every Node service has
  this. It works; it's not elegant.

### 3.4 When to pick Node/TS for this kind of problem

- **Service whose traffic is I/O-bound and whose hot path is
  not the limiter.** A 2 kLoC TypeScript service that mostly
  proxies / transforms HTTP is the sweet spot. The 200-LoC
  rate limiter is a small part of a larger whole.
- **Team that already runs Node and is more productive in
  TypeScript than in Go or Rust.** Type safety without
  ceremony.
- **Prototype / MVP where the time-to-first-request matters
  more than the p99 latency.** Node is the fastest to
  scaffold.

## 4. Concepts you should be able to answer after this project

### Q1 · What problem does this project solve and why does it matter?

A **Token-Bucket Rate Limiter** solves "how to protect a service
from being overwhelmed by any single client, while still allowing
short bursts of legitimate traffic." It matters because:

- **Unbounded traffic is a denial-of-service.** Even an honest
  client with a retry loop can saturate your service if every
  request succeeds. A rate limiter turns "every request is a
  success" into "you have 10 requests; come back in 500 ms."
- **The token-bucket variant specifically allows bursts.** A
  fixed-window limiter (e.g. "10 requests per minute, reset on
  the minute boundary") punishes the client who needs 10
  requests at 12:00:59 and gets blocked until 12:01:00. The
  token-bucket algorithm is *continuous* — the bucket refills
  smoothly, so a client who has been quiet gets the full
  capacity available when they return.
- **It's a load-shedding primitive.** Modern service meshes
  (Envoy, Istio) implement rate limiters as the first line of
  defense. Building one from scratch is how you understand what
  the mesh is doing for you.

### Q2 · What is the reference architecture for an in-memory rate limiter?

The architecture every implementation in this project follows:

```
            ┌──────────────────────────────────────────────┐
   request  │   ┌────────────────┐                        │
   ────────▶│──▶│  Resolve Key   │ (IP, user, API key)    │
            │   └───────┬────────┘                        │
            │           │                                 │
            │           ▼                                 │
            │   ┌────────────────┐                        │
            │   │  Lookup bucket │◀─── HashMap<key, …>    │
            │   │  in map        │                        │
            │   └───────┬────────┘                        │
            │           │  (key absent → create full)     │
            │           ▼                                 │
            │   ┌────────────────┐                        │
            │   │  Lazy refill   │  tokens += dt × rate   │
            │   │  on demand     │  (capped at capacity)  │
            │   └───────┬────────┘                        │
            │           │                                 │
            │           ▼                                 │
            │   ┌────────────────┐                        │
            │   │  Allow or deny │  if tokens >= 1: allow │
            │   │                │  else: 429 + Retry-After│
            │   └───────┬────────┘                        │
            │           │                                 │
            │   ┌───────┴────────────────────────┐        │
            │   │                                │        │
            │   ▼                                ▼        │
            │ Stamp X-RateLimit-*       next.ServeHTTP     │
            │ on every response         (allow path)      │
            │                                           │
            │  ┌────────────────────────────────────┐     │
            │  │  Background sweep every N seconds: │     │
            │  │  delete buckets idle > T           │     │
            │  └────────────────────────────────────┘     │
            └──────────────────────────────────────────────┘
```

Three properties define the reference architecture:

1. **Lazy refill on demand.** No background ticker per bucket.
   The refill math runs when the bucket is read, using the
   `now() - last_refill` delta. This is what makes the algorithm
   O(1) per request and unbounded in keyspace.
2. **Bounded memory via a periodic sweep.** The map would grow
   forever otherwise. The sweep runs on a fixed cadence and
   removes buckets that haven't been touched in T (typically
   1 hour).
3. **Single critical section for read-modify-write.** A
   request for IP X must observe, decide, and (if allowed)
   consume in a single lock acquisition, otherwise two
   concurrent requests could both see "1 token left" and both
   succeed.

### Q3 · Why pick Go, Rust, or Node for this case? (use the data, not opinions)

| Need | Go wins | Rust wins | Node wins |
|---|---|---|---|
| Cold start ≤ 10 ms | ✅ ~5 ms | ❌ ~10-20 ms | ❌ ~200 ms |
| Image size ≤ 20 MB | ✅ ~13 MB | ✅ ~5-15 MB | ❌ ~50 MB |
| Compile-time safety | partial (type system) | full (ownership + types) | partial (TS types) |
| Concurrency safety evidence | `go test -race` | `clippy -D warnings` (no runtime check) | model-implied |
| Ecosystem (HTTP server) | stdlib | axum/tower | express/fastify |
| Team ramp-up | days | weeks | hours |

**Recommendation**:
- **Go** is the right pick for a *long-running service* that
  needs fast cold start, small image, and proven concurrency
  safety. It's the median of all three.
- **Rust** is the right pick if the rate limiter is *part of a
  larger system* where compile-time safety pays off across many
  services. For a 200-line standalone service, the compile
  time is not worth it.
- **Node/TS** is the right pick if the limiter is *a small part
  of a Node service* whose hot path is something else (DB,
  cache, RPC). The TypeScript ergonomics and the team velocity
  win.

### Q4 · What real tradeoffs emerged in each language?

- **Go**: The stdlib `net/http` is the secret weapon, but
  trusting `X-Forwarded-For` is left to the developer — and the
  developer needs to *know* it's a security decision, not a
  config knob.
- **Rust**: `std::sync::Mutex` is the right choice for
  synchronous critical sections, but the test suite
  (`#[ignore]`d) is the proof that *correctness tooling is not
  yet as polished as the language*. The 1.81 Rust pin in the
  Dockerfile is a real constraint.
- **Node/TS**: Single-threaded by construction, but also
  *single-process by implication* — which is the limit nobody
  thinks about until they deploy with 4 PM2 workers and
  discover 4× the tokens.

### Q5 · How would you scale this from 100 to 1M users?

Three jumps:

1. **100 → 10 k users**: The current in-memory impls suffice.
   The bottleneck will be the single mutex (Go/Rust). Add a
   benchmark; if p99 climbs above 5 ms, shard the map by
   `fnv32(key) % 256` and lock per shard. Node is fine until
   the event loop saturates (likely ~50 k RPS on a single core).
2. **10 k → 100 k users**: The bottleneck becomes CPU
   contention on a single box. Scale horizontally: N
   stateless replicas behind a load balancer, each with its
   own in-memory map. The effective rate becomes `N × C`
   tokens per IP — which is the bug you accept. To fix it,
   move state to a shared store.
3. **100 k → 1 M users**: Move state to **Redis** (or any
   shared KV). The token-bucket algorithm is just `INCR +
   EXPIRE` or a Lua script for atomicity. The "scale-out"
   pattern is well-trodden; the `rate-limiter-flexible` npm
   package implements it.

At 1 M users, the rate limiter is no longer a 200-line
component — it's a distributed system problem. The current
implementations are the *single-process* building block; the
*distributed* one is a future project.

### Q6 · What anti-patterns did you find and fix?

The review surfaced 27 issues. The ones that recurred across
multiple impls (and are worth fixing in the *next* cycle as
project-wide conventions):

- **Implicit trust in client IP.** Every impl either ignores
  `X-Forwarded-For` (Go) or trusts it without a hop count
  (Node). The fix is a hop-count or CIDR-list config.
- **`#[ignore]`d tests that cover safety properties.** The
  Rust impl ignored the one test that would catch a race.
  The fix is to make the gate independent of the test harness
  (`cargo test -- --ignored` in CI).
- **Unbounded `Map<key, state>`.** Every impl is vulnerable
  to a key-spraying DoS. The fix is a soft cap (1 M buckets
  → log + drop new keys).
- **Cross-language behavior drift.** `/status` updates
  `last_seen` in Go, doesn't in Rust, doesn't mutate in Node.
  Pick one and document it in the spec.

## 5. Takeaways for the journal

These are the durable, reusable generalizations that survived
this cycle:

- **The `Clock` trait/interface is a universal pattern for
  testable time-dependent code.** All three languages converged
  on the same design. The shape varies (struct, trait, function)
  but the abstraction is the same.
- **`std::sync::Mutex` (Rust) / `sync.Mutex` (Go) /
  single-threaded (Node) are different answers to the same
  question.** The right answer is a function of the *critical
  section length*, not the language. Short critical sections
  prefer sync; long ones prefer async.
- **"Client IP" is a trust-boundary decision, not a fact.** Every
  impl that touches the network has to make it; the spec
  doesn't pin it down; the result is a config knob that is
  either too lax (Node) or absent (Go).
- **A test you don't run is a bug you don't catch.** The Rust
  `#[ignore]` is a real cost. The fix is to make the gate
  independent of the harness.
- **A test that verifies the *relationship* between two outputs
  catches more bugs than a test that verifies each output
  independently.** (`Retry-After` header == `retry_after_seconds`
  body field.)
- **The "pure core, thin shell" pattern is at its most natural
  in TypeScript** because the language's class privacy and the
  `class`/`function` separation are first-class. The same
  pattern works in Go (subpackage) and Rust (lib + bin) but
  with more ceremony.
- **In-memory maps keyed by user input are a DoS surface.**
  Every cache needs a bound.

These are captured in `learning_journal.md` for the next cycle.
