## 1. Go — `go-impl/`

### 1.1 Idioms used

- **32-way sharded `sync.Mutex`, each guarding its own `map[string]*ClientBucket`.** A client key
  is hashed with FNV-1a and reduced modulo 32 (a bitmask, since 32 is a power of two) to pick a
  shard. This scales lock contention down by ~32x versus a single mutex over one big map, at the
  cost of one hash per request. The single-mutex version came first (correct for "10s of clients,
  low RPS" per the spec); sharding was added once the design needed to handle more concurrent
  distinct clients.
- **Package-level `Clock` interface** (`interface { Now() time.Time }`) injected at construction.
  Default `RealClock{}`; tests use a `fakeClock` with an `Advance(d)` method. One-method interface
  — the idiomatic Go "small interface" pattern.
- **`slog` with `NewJSONHandler`** for structured logging, threaded through `loggingHandler`.
- **Middleware as `func(http.Handler) http.Handler`**, composed via
  `mux.Handle("/", rl.Middleware(serveWelcome))` — the same shape used across the Go HTTP
  ecosystem (chi, gin, stdlib).
- **`signal.NotifyContext(ctx, syscall.SIGINT, syscall.SIGTERM)`** for shutdown signaling, paired
  with `srv.Shutdown(shutdownCtx)` — the standard graceful-shutdown sequence.
- **Manual env parsing with `slog.Warn` fallback** — a misconfigured deployment logs a warning and
  starts with defaults rather than refusing to boot.

### 1.2 Strengths in this domain

- **`net/http` is production-grade out of the box.** `ReadHeaderTimeout`, `ReadTimeout`,
  `WriteTimeout`, `IdleTimeout`, and `Shutdown` are all there — no third-party framework needed.
- **`go test -race` is the killer feature for a rate limiter specifically.** It will catch a
  missed mutex acquisition or a copied struct on the first CI run. Neither Rust's `#[ignore]`d
  loom-style test nor Node's single-threaded model gives you an equivalent automatic check.
- **A dedicated concurrency test suite exists and runs**: `TestRateLimiter_ConcurrentAccessNoRace`,
  `TestRateLimiter_ConcurrentDifferentKeys`, `TestMiddleware_ConcurrentRequests` — these are not
  disabled, unlike Rust's equivalent (see §2.3).
- **Compilation is a soft specification.** Type mismatches in middleware composition fail at
  `go build`, not at request time.

### 1.3 Weaknesses in this domain

- **The error-type story is "values, not enums".** `error` is an interface; you can't exhaustively
  match on it the way Rust matches on `Result<T, E>`. `errors.Is`/`errors.As` help but are less
  ergonomic.
- **A pluggable client-key strategy (`ClientKeyStrategy`, `ForwardedHeaderKeyStrategy`) was built
  but never wired into the running server** (`code_review.md` XLANG-MAJOR-001, GO-MAJOR-001) — the
  live code path still only reads `r.RemoteAddr`, so this Go implementation has no working path to
  trust a reverse proxy's `X-Forwarded-For` today, despite the code existing to do it.
- **Mutex-protected fields are a convention, not a compiler-enforced contract** — `idleTTL` is
  correctly guarded today (readIdleTTL()/SetIdleTTL()), but nothing stops a future method from
  reading the raw field directly (GO-MAJOR-002).

## 2. Rust — `rust-impl/`

### 2.1 Idioms used

- **16-way sharded `std::sync::Mutex<HashMap<IpAddr, ClientBucket>>`.** Same pattern as Go's
  sharding, fewer shards (16 vs 32) because Rust's `HashMap` carries more per-shard overhead than
  Go's map, so the crate trades a few more collisions for less memory duplication across shards.
- **`Arc<dyn Clock>` for time injection**, with `SystemClock` in production and `MockClock` (with
  an explicit `.advance()`) in tests.
- **A trait-based `ClientKeyStrategy` was designed** (`client_key.rs`) **but never wired into the
  middleware/handlers**, which instead use axum's `ConnectInfo<SocketAddr>` extractor directly.
  Functionally equivalent to the Go situation — the abstraction layer is dead code.
- **`axum` + `tower` middleware via `route_layer(from_fn_with_state(...))`**, applied only to the
  `/` route (registered before `route_layer`), leaving `/status` (registered after) unaffected —
  this is the axum idiom for "middleware on a subset of routes".
- **`start_instant` + `start_system_time` pair, captured once at construction**, used to convert a
  monotonic `Instant` into a wall-clock Unix-epoch second deterministically under `MockClock`.

### 2.2 Strengths in this domain

- **The type system encodes the Allowed/Denied distinction as an enum** (`Decision::Allowed { .. }`
  / `Decision::Denied { .. }`), so a caller cannot forget to check which branch they're in — unlike
  Go's `Decision` struct where `Allowed` is just a `bool` field a caller could ignore.
- **`std::sync::Mutex` over `tokio::sync::Mutex` for a synchronous critical section** — correctly
  avoids the cost of an `.await` and a runtime wake-up notification for work that's a hash lookup
  plus float arithmetic.
- **Saturating float→int casts** (post Rust 1.45) mean `x as u64` on a NaN or out-of-range `f64`
  can't be undefined behavior, unlike C/C++. This is a genuine safety net, though `code_review.md`
  RUST-MINOR-001 notes the current code relies on this implicitly rather than checking explicitly.

### 2.3 Weaknesses in this domain

- **The one test that exercises real concurrent load (50 tokio tasks hitting the same IP) is
  `#[ignore]`d** because the multi-threaded tokio test runtime hangs intermittently at teardown in
  this environment. The property is still asserted by synchronous tests (which prove the lock spans
  refill+check+consume), but that's a proof by construction, not a proof under actual concurrent
  execution. `code_review.md` RUST-MAJOR-001 suggests a fix: test the same property with
  `std::thread::scope` instead of tokio tasks, since `RateLimiter::check` itself has no `.await` in
  it and doesn't need an async runtime to be exercised concurrently.
- **Config parsing fails silently to defaults** (`unwrap_or(default)` on every env var) — an
  operator typo in `RATE_LIMITER_CAPACITY` produces no log line at all, the quietest of the three
  implementations' failure modes.

## 3. Node/TypeScript — `node-impl/`

### 3.1 Idioms used

- **Pure class + Express adapter** (`rateLimiter.ts` has zero HTTP imports; `index.ts` wires it to
  Express). The class takes a `clock: () => number` and returns plain data.
- **`zod`-validated config that hard-fails boot** (`ConfigError`) on the first batch of invalid
  env vars, using `z.flattenError` to report all problems at once rather than one at a time.
- **Pre-allocated 429 response body** (string concatenation instead of `JSON.stringify` on a fresh
  object) as a hot-path optimization, since under sustained overload the 429 path can be the
  majority of responses.
- **`cleanupTimer.unref()`** so a pending cleanup interval never prevents the Node process from
  exiting — a common Node production gotcha ("why won't my server exit after Ctrl-C?").
- **A `ClientKeyStrategy`-shaped abstraction was built** (`clientKeyStrategy.ts`) **but `index.ts`
  duplicates the same logic inline instead of importing it** — the third occurrence of the same
  dead-abstraction pattern seen in Go and Rust (see §5 cross-cutting takeaway).

### 3.2 Strengths in this domain

- **No shared-memory race is possible in-process** — Node's single-threaded event loop means the
  `Map` mutations in `tryConsume` are trivially atomic with respect to other requests, as long as
  the method stays synchronous (it does). This sidesteps an entire class of bugs Go and Rust have
  to actively defend against with mutexes.
- **The strictest config validation of the three** — `zod` + hard-fail-on-boot means a
  misconfigured Node deployment never starts serving traffic with silently-wrong limits, unlike Go
  (warns, continues) or Rust (silently falls back).
- **Mutation-tested, not just coverage-tested**: `docs/mutation_gate.md` reports a Stryker score of
  71.05% against a 60% gate (PASS), re-verified current in this review pass. Neither Go nor Rust has
  an equivalent mutation-testing gate yet.

### 3.3 Weaknesses in this domain

- **"Single-threaded" only guarantees safety within one process** — running N replicas (PM2
  cluster, Kubernetes horizontal scaling) gives each replica its own independent `Map`, so the
  effective global capacity becomes `10 × N`, not `10`. Not documented in the README today
  (`code_review.md` NODE-MINOR-002).
- **`TRUST_PROXY` is a single boolean**, not a hop-count or CIDR allowlist — the same "all or
  nothing" trust-boundary problem as Go's missing trust-proxy support, just inverted (Node has the
  feature but it's too permissive; Go lacks the feature entirely).
- **The 500-error path is an empty `it.todo`** — real production logic (don't leak stack traces,
  return JSON, check `res.headersSent`) with zero test coverage.

## 4. Conceitos que você deve saber responder (6 perguntas pedagógicas)

### Q1 · Why lazy refill instead of a background timer per bucket?

A per-bucket timer means N buckets → N scheduled callbacks, which is O(N) background work even
for clients who never send another request. Lazy refill computes
`tokens = min(capacity, tokens + elapsed_since_last_request × rate)` only when a request actually
arrives, so idle clients cost zero CPU between requests. The trade-off is that every read must
also refill first — you can never look at `tokens` without accounting for elapsed time since the
last touch.

### Q2 · Why does every implementation inject a `Clock` instead of calling the system clock directly?

Because the lazy-refill formula depends on elapsed wall-clock time, and tests need to assert exact
token counts after "1 second passes" without an actual `sleep(1)` (slow, flaky under CI load). All
three implementations converge on the same seam — a swappable clock — expressed differently per
language (Go: one-method interface; Rust: `Arc<dyn Clock>` trait object; Node: a plain function
type `() => number`). This is a reusable pattern for any time-dependent logic: TTLs, debounce,
retry backoff, idle eviction.

### Q3 · Why is a per-key sharded mutex (Go, Rust) preferable to one global mutex here — and why is neither approach needed in Node?

Every request both reads and writes the bucket for its key, and the key space churns as new
clients connect — the worst-case access pattern for read-optimized structures like Go's `sync.Map`.
A single mutex over one big map serializes all requests regardless of key; sharding (hash the key,
pick 1-of-N independent mutex+map pairs) keeps the same simple critical section while letting
requests for different clients proceed in parallel. Node doesn't need any of this because
JavaScript's single-threaded event loop means no two requests can interleave their `Map` mutations
in the first place — concurrency-safety is a property of the runtime, not something the code has
to construct.

### Q4 · What does "the mutex must span refill + check + consume" actually protect against, concretely?

If refill and the allow/deny check were two separate lock acquisitions, two concurrent requests for
the same client could both refill-and-read "1.5 tokens available", both decide "allowed", and both
decrement — over-consuming the bucket by one token beyond its capacity. The critical section must
cover the full read-modify-write cycle atomically. This is exactly what Go's `TestMiddleware_ConcurrentRequests`
and Rust's synchronous tests assert (though Rust's true concurrent-load test is currently disabled
— see `code_review.md` RUST-MAJOR-001), and it's the property a "TOCTOU" (time-of-check to
time-of-use) bug would violate if someone "optimized" the lock scope without understanding why it's
there.

### Q5 · How would this design change from 10s of clients to 1M+ concurrent clients?

1. **10s → 10,000s of distinct clients on one box**: sharding (already implemented in Go/Rust) is
   the right answer — it buys headroom without changing the algorithm. Node scales differently:
   the bottleneck becomes the single event loop's throughput, not lock contention, since there's no
   lock to begin with.
2. **10,000s → 100,000s of clients**: a single process/box runs out of CPU. Scale horizontally with
   N stateless replicas behind a load balancer — but each replica has its own map, so the effective
   rate limit becomes `N × capacity` per client unless requests are sticky-routed by client key.
   This is a real, currently-undocumented limitation in all three implementations
   (`code_review.md` NODE-MINOR-002 flags it for Node; it applies equally to Go/Rust).
3. **100,000s → millions**: move bucket state to a shared store (Redis `INCR`+`EXPIRE`, or a Lua
   script for atomicity). At that point the rate limiter stops being a 200-line in-process
   component and becomes a distributed-systems problem — the three implementations here are the
   single-process building block, not the final answer at that scale.

### Q6 · What anti-patterns did this review actually find, re-verified against the current code?

- **An abstraction built and never wired in, in all three languages** (`ClientKeyStrategy` in Go,
  `client_key.rs`'s trait in Rust, `clientKeyStrategy.ts` in Node) — each has its own dedicated unit
  test achieving high/100% coverage, which is precisely why it slipped past prior review passes:
  a well-tested file isn't the same as a *used* file. Check for callers, not just tests, before
  trusting a seam is live.
- **Trust in client-supplied IP information handled three inconsistent ways**: Go has no working
  path to trust a proxy at all; Node has an all-or-nothing boolean; Rust has neither the feature
  nor a documented gap. None of the three lets an operator scope trust to "this many hops" or
  "these CIDRs only".
- **A concurrency-safety test disabled in Rust** because of test-harness flakiness, with no
  currently-running equivalent — a lower-stakes concurrency primitive (OS threads instead of tokio
  tasks) would likely sidestep the flakiness without losing the coverage (`code_review.md`
  RUST-MAJOR-001).
- **Cross-implementation behavior drift on an edge case the spec doesn't pin down**: whether
  hitting `/status` counts as "activity" for idle-eviction purposes. All three made a different,
  silent choice. This is a spec gap, not three independent bugs — the fix belongs in `docs/spec.md`.

## 5. Takeaways for the journal

- **The `Clock` trait/interface/function-type is a universal pattern for testable time-dependent
  code.** All three languages converged on the same design; the shape varies, the abstraction
  doesn't.
- **Sharded-mutex vs. single-mutex vs. single-threaded are three answers to the same access-pattern
  question**, not three arbitrary language defaults — the right answer depends on read/write ratio,
  key churn, and (for Node specifically) whether the runtime already rules out interleaving.
- **"Client IP" is a trust-boundary decision, not a fact.** Every implementation that talks to the
  network over HTTP has to make this decision explicitly (hop count / CIDR allowlist), and none of
  the three currently does so completely.
- **A well-tested abstraction is not the same as a used abstraction.** Grep for callers before
  trusting that an interface/trait/class is load-bearing in production. This generalizes beyond
  this project: dead code with its own passing tests is a specific, sneaky failure mode of
  "we value test coverage" cultures.
- **A concurrency test you don't run is a bug you don't catch** — if a test is disabled for
  test-harness reasons, look for a simpler concurrency primitive that avoids the harness problem
  before accepting the gap.
- **Mutation testing (Node's Stryker gate, 71.05%) catches what coverage can't**: a line can be
  executed by a test without the test asserting anything meaningful about its behavior. Worth
  extending to Go/Rust in a future cycle.
