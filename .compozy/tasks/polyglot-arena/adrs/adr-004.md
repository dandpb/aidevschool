# ADR-004: Prediction + outcome storage — dedicated `learner/predictions.yaml`

## Status

Accepted

## Date

2026-06-25

## Context

The hard prediction gate (ADR-002) requires recording, per arena run, the
learner's predicted winner for each of latency / memory / throughput, the
measured winner, and the gap — durably and queryably, so the V2 calibration meter
can compute per-metric accuracy over time.

The canonical `learner/learning_state.yaml` records spaced-repetition reviews as
`units_log[*].reviews[*]` with `rating` (again|hard|good|easy) and `gate_outcome`
(fail|pass_*). It has **no field for a prediction**, and its schema
(`learner/substrate/schema.yaml`) is validated and consumed by the substrate that
regenerates four downstream artifacts (codexDojo `learner.ts`, mavis snapshot,
minimaxDojo whiteboard, pixelDojo review slice). Adding a prediction field there
is a breaking schema change with four blast-radius consumers.

## Decision

Store predictions in a **dedicated, structured `learner/predictions.yaml`** —
independent of `learning_state.yaml`, append-only, one record per (project,
metric, run). The substrate is extended to read it and expose a small
`predictions` summary on `LearnerSnapshot` for the dashboard. The
`learning_state.yaml` schema is left untouched.

Record shape (per metric):

```yaml
predictions:
  - project: "01_rate_limiter"
    run: "2026-06-25T18:00:00Z"   # passed in; clock is not read inside the run
    metric: latency               # latency | memory | throughput
    predicted: rust               # go | rust | node
    actual: rust
    correct: true
    reason: "ownership avoids GC pauses"   # optional, one line
```

## Alternatives Considered

### Alternative 1: Extend `units_log[*].reviews` with a `prediction_metrics` field

- **Description**: Add an optional per-metric prediction block to the existing
  review records.
- **Pros**: One state file; predictions co-located with the gate outcome.
- **Cons**: Breaking change to a validated schema with four downstream consumers;
  couples a fast-iterating new vocabulary to the stable FSRS review record.
- **Why rejected**: Blast radius and coupling outweigh co-location; the prediction
  vocabulary is not yet stable.

### Alternative 2: Append predictions to `learner/journal.md`

- **Description**: Log each prediction as a tagged free-text journal entry.
- **Pros**: Zero schema work; non-breaking.
- **Cons**: Unstructured; computing per-metric accuracy for the V2 meter means
  parsing prose. Defeats the calibration goal.
- **Why rejected**: V1 explicitly chose per-metric predictions to seed calibration;
  the data must be structured.

## Consequences

### Positive

- Non-breaking: `learning_state.yaml` and its schema are unchanged.
- Structured + queryable: per-metric accuracy is a trivial group-by; the V2 meter
  reads this file directly.
- Clear ownership boundary: prediction calibration is its own concern, separate
  from spaced-repetition mastery state.

### Negative

- A second learner-state file to keep consistent (mitigated: it is append-only and
  independent).
- The substrate gains one more input to read and one more snapshot field to
  render.

### Risks

- **Drift between the two files** → low; `predictions.yaml` references project +
  run and never mutates `learning_state.yaml`.
- **Determinism** (no `Date.now()` inside runs per repo rules) → the `run`
  timestamp is supplied by the command invocation, not read inside arena logic.

## Implementation Notes

- File: `learner/predictions.yaml`, top-level `predictions: [ … ]`, append-only.
- Substrate: read in `learner/substrate/`, add `predictions: {count, byMetric:
  {latency,memory,throughput: {correct,total}}}` to `LearnerSnapshot` (small,
  additive; dashboard can ignore until V2 meter UI exists).
