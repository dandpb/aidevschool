# ADR-005: Agent topology — fairness-auditor + arena-narrator, verifier reused

## Status

Accepted

## Date

2026-06-25

## Context

V1's primary success bar (PRD G1) is **trustworthy lessons**: every comparison
fair, every narrative claim verifier-confirmed, zero wrong lessons. The council's
top risk is that one agent writing all three implementations *and* declaring a
winner measures the agent's per-language effort, not the languages — a confident,
wrong lesson the learner cannot catch. This double-violates producer ≠ verifier
(the agent benchmarks its own code and narrates why its own result happened).

Existing agents (engine `.claude/agents/`): `dev-go` / `dev-rust` / `dev-node`
(producers, full tools), `reviewer` / CRITICO (cross-language code review),
`verifier` / PROMĘTOR (adversarial, **no Write/Edit**, opus, re-derives from
scratch). Agents are Markdown files with frontmatter (`name`, `description`,
`tools`, `model`, `color`).

## Decision

Add exactly **two** new agents and reuse the existing verifier; wire them so no
agent judges its own output.

1. **`fairness-auditor`** (new, judge). Frontmatter: no Write/Edit (Read, Grep,
   Glob, Bash only), `model: opus`. Runs **before** benchmarking. Checks each of
   the three impls against a written effort budget (idiomatic, not hand-tuned),
   flags unfair handicapping/tuning, and emits a pass/flag verdict. A flag blocks
   the benchmark stage.
2. **`arena-narrator`** (new, producer). Frontmatter: Write/Edit allowed,
   `model: opus`. Consumes the scoreboard (`aggregated.json`) + CRITICO's code
   study and drafts `arena_report.md` — per-metric winner + grounded "why" + one
   transferable concept. May assert only what the evidence supports.
3. **`verifier`** (reused, PROMĘTOR). After narration, re-checks **every**
   narrative claim against `aggregated.json` and the diff. Any unsupported claim
   blocks the prediction gate (the report cannot reveal).

Pipeline order (orchestrated by `/devschool-arena`): dev-{go,rust,node} →
**fairness-auditor** → benchmark_runner + analyzer → reviewer (CRITICO code
study) → **arena-narrator** → verifier (PROMĘTOR) → prediction gate → reveal.

## Alternatives Considered

### Alternative 1: One combined "arena agent" does impls + benchmark + narrative

- **Description**: A single agent produces everything.
- **Pros**: Fewer files; simplest wiring.
- **Cons**: Producer = verifier; exactly the epistemic failure the council
  flagged. No independent check on fairness or narrative truth.
- **Why rejected**: Violates the primary success bar (G1) by construction.

### Alternative 2: Reuse CRITICO as the fairness auditor too

- **Description**: Let the code reviewer also judge effort-budget fairness.
- **Pros**: No new agent.
- **Cons**: CRITICO also feeds the code study that the narrative cites; having it
  gate fairness blurs producer/judge roles and overloads one agent across two
  pipeline stages with different independence needs.
- **Why rejected**: Fairness auditing must be an independent pre-benchmark gate; a
  dedicated thin judge keeps roles clean. (Low cost: one small agent file.)

## Consequences

### Positive

- Producer ≠ verifier holds at both risk points: impl fairness (auditor) and
  narrative truth (verifier), neither judging its own output.
- Reuses the existing adversarial verifier and its no-Write discipline.
- Minimal additions: two agent files in the proven format.

### Negative

- Two more agents to maintain and two more pipeline stages (latency per run
  increases).
- The effort-budget rubric the auditor checks is not yet written (see Open
  Questions in the PRD/TechSpec).

### Risks

- **Auditor false-confidence** (passes an unfair impl) → mitigated by the
  learner-reproducible runs in ADR-003 (the bench can be re-run with perturbed
  inputs to sanity-check rankings).
- **Effort-budget subjectivity** → start with a written per-language rubric;
  refine as projects accumulate.
