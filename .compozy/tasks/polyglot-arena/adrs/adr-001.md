# ADR-001: Polyglot Arena V1 — narrative-led projection with a prediction gate

## Status

Accepted

## Date

2026-06-25

## Context

The Polyglot Comparison Arena is a learning lab: AI agents implement each
curriculum challenge in Go, Rust, and Node, and the solo-dev learner studies the
output. The initial framing was "agents code all, I study" with three co-equal
deliverables per run (teaching narrative, benchmark scoreboard, annotated code
study).

Two facts reshaped the scope:

1. **~70% of the infrastructure already exists.** A working benchmark harness
   (`curriculum/01_rate_limiter/benchmarks/run_matrix.sh` + `analyze_results.py`,
   shared `_shared/benchmarks/analyzer.py` with an N≥3 / CV%<20% gate), real
   Go/Rust/Node code for all 18 projects, the producing agents (CRITICO review,
   GALILEU benchmarks, PROMĘTOR verifier), and the evidence→dashboard pipeline
   (`learner/learning_state.yaml` → substrate → codexDojo). Net-new is narrow.
2. **A prior standalone `engines/polyglotEvolutionArena/` was demoted
   (2026-06-21)** precisely because it was a new engine shell with no executable
   seam into the shared pipeline.

A four-advisor council (pragmatic-engineer, architect-advisor, product-mind,
devils-advocate) converged on three load-bearing conclusions:

- The three sections are **not co-equal**: the teaching narrative is the hero
  and the only genuinely net-new content; the scoreboard and code study are its
  grounding evidence.
- **Pure passive study violates the ecosystem's learning gate** ("the learner
  attempts and is evaluated on executable evidence before mastery"). Reading
  auto-generated reports produces recognition, not mastery.
- **Agent-authored benchmarks are epistemically unsafe**: one agent writing all
  three implementations measures *that agent's per-language effort*, not the
  languages. A naive Node impl vs. a tuned Rust impl yields a confident, wrong
  lesson — and the learner (by design lacking the skill) is the worst-positioned
  person to catch it. This double-violates producer ≠ verifier (the agent
  benchmarks its own code and narrates why its own result happened).

## Decision

Build V1 as a **narrative-led projection over the existing substrate, gated by a
learner prediction**, not a new engine and not three co-equal artifacts.

1. **Architecture — projection, not engine.** Ship a `/devschool-arena` command
   plus a thin module under `curriculum/_shared/arena/`. The arena orchestrates
   the existing loop parametrized by language and aggregates; GALILEU benchmarks,
   CRITICO reviews, PROMĘTOR verifies. No new `.claude/`, no fourth engine.
2. **Artifact — aggregation/view, not duplication.** Per-language runs still emit
   the canonical `curriculum/NN/docs/{benchmark_results,code_review,evolution_report}.md`
   (filesystem stays source of truth). The arena emits one generated
   `curriculum/NN/docs/arena_report.md` that links/summarizes them and adds the
   single net-new content: the teaching narrative. Language is an axis
   (frontmatter/path), not three parallel doc trees.
3. **Narrative is the hero, and it is grounded.** The narrative may only assert
   what the scoreboard numbers and CRITICO's diff support; winner + "why" must
   cite the measured metric. One crisp transferable concept per run.
4. **Prediction gate (active recall) is in V1, not deferred.** Before the report
   unlocks, the learner commits a single prediction ("which language wins on
   latency, and why?"). The logged prediction + outcome is the executable-ish
   evidence the learning gate requires, and the guess-vs-measured gap is the
   learning moment. This is the value core, not scope creep.
5. **Benchmark-fairness controls are mandatory.** Each implementation carries a
   written effort budget (idiomatic, not hand-tuned); a *different* agent audits
   each impl for unfair handicapping before the bench runs; the benchmark is
   reproducible by a committed runner the learner can execute and perturb (change
   N, input size, flags) and watch the ranking move. The run, not the narrative,
   is the authoritative artifact.
6. **The one V1 reusable seam: de-hardcode the runner.** Extract a
   `run_benchmark(project_dir, lang)` contract from the project-01 hardcode,
   reusing `analyzer.py` unchanged, validated against 01's known-good output as a
   regression baseline. Prove the pipeline on **2–3 projects, not all 18.**

## Alternatives Considered

### Alternative 1: Three co-equal deliverables, agents code all, learner studies (original framing)

- **Description**: Each run produces narrative + scoreboard + code study as
  equally-weighted first-class sections; learner reads them.
- **Pros**: Matches the learner's initial stated preference; most "complete" per
  run.
- **Cons**: Mis-allocates effort (over-builds the two easy sections, under-builds
  the one hard/risky one); ships a read-once artifact with weak retention;
  violates the learning gate; ignores benchmark-fairness risk.
- **Why rejected**: Council unanimous — co-equal treatment hides where V1 fails
  (a plausible-but-wrong narrative) and produces edutainment, not skill change.

### Alternative 2: Generic multi-language benchmark framework as a V1 prerequisite

- **Description**: Build harness autodiscovery, container matrices, and
  per-language config schemas so any project/language runs out of the box.
- **Pros**: Would scale to all 18 projects × 3 languages cleanly.
- **Cons**: A V3 platform masquerading as a V1 prerequisite; large net-new
  surface; re-creates the over-engineering that got the prior design demoted.
- **Why rejected**: The architect flagged it as a trap. V1 needs only the
  `run_benchmark(project_dir, lang)` seam proven on a few projects.

### Alternative 3: Standalone arena engine

- **Description**: A fourth engine root with its own `.claude/`, agents, and loop.
- **Pros**: Conceptual isolation.
- **Cons**: Exactly the structure demoted on 2026-06-21; no executable seam into
  the shared pipeline; duplicates the 5-phase loop already in
  miniMaxEvolutionEngine.
- **Why rejected**: Repeats a settled mistake.

## Consequences

### Positive

- Smallest net-new build: de-hardcoded runner + one report template + a grounded
  narrative generator + a lightweight prediction log. Reuses benchmark harness,
  agents, and dashboard pipeline untouched.
- Honors the learning gate and producer ≠ verifier without new infrastructure.
- The differentiator (narrative + scoreboard + diff + prediction as one artifact)
  is preserved and is exactly the market gap incumbents miss.
- Compounds: the prediction log becomes a calibration track-record over runs.

### Negative

- The learner's original "purely passive study" preference is partially
  overridden by the prediction gate (a small required interaction per run).
- Narrative quality is capped to "grounded and correct" over "eloquent" in V1.
- Only 2–3 projects are covered at V1; the other 15 are content, deferred.

### Risks

- **Unfair benchmarks teaching the wrong lesson** (highest) → mitigated by effort
  budgets + adversarial impl audit by a second agent + learner-reproducible runs.
- **Narrative confabulation** → mitigated by PROMĘTOR verifying narrative claims
  against the numbers before the learner sees it (producer ≠ verifier).
- **Runner generalization scope-creep** → mitigated by extracting only the
  `run_benchmark(project_dir, lang)` contract, validated against project 01.

## Implementation Notes

- V1 surface: `/devschool-arena <project>` → runs/aggregates → writes
  `curriculum/NN/docs/arena_report.md` → records prediction + outcome into
  `learner/` → `python3 -m learner.substrate` refreshes the dashboard.
- Defer to V2+: all-18 coverage, cross-project comparison matrix ("which language
  fastest in category Y across projects"), the "fix the slow impl to beat the
  bench" graded-challenge mode, narrative style/polish.
