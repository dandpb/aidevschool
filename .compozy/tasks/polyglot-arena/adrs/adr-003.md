# ADR-003: Arena orchestration — `/devschool-arena` command + extracted runner seam

## Status

Accepted

## Date

2026-06-25

## Context

ADR-001/002 set the arena as a projection over existing substrate, gated by a
per-metric prediction. Codebase exploration found:

- `curriculum/_shared/benchmarks/analyzer.py` is complete and reusable
  (`BenchmarkAnalyzer.analyze_raw_samples` / `validate_report` / `export_json`,
  with the CV<20% and N≥3 gates).
- The k6 + docker-stats runner exists but is **hardcoded to project 01**
  (`run_matrix.sh`, `generate_report.py`): image names (`rl-go`/`rl-rust`/
  `rl-node`), host ports (incl. a dev-machine conflict override 18080), container
  names, absolute paths, and a static per-language LOC/image-size dict.
- The engine already runs producer/verifier agents via `.claude/commands/devschool/*.md`
  commands and `.claude/agents/*.md` agents in a proven format.

The decision is how the arena is orchestrated and how the runner is made reusable
without rebuilding the engine or over-generalizing into a framework (the trap that
demoted the prior design).

## Decision

1. **Command-driven orchestration.** Add
   `engines/miniMaxEvolutionEngine/.claude/commands/devschool/arena.md`
   (`/devschool-arena <project>`), in the existing command format. It sequences:
   produce/confirm the three impls → fairness audit → benchmark → analyze →
   code-study review → narrate → verify → prediction gate → reveal + substrate
   refresh.
2. **Thin Python module.** Arena glue lives at `curriculum/_shared/arena/`
   (orchestration helpers, report assembly, prediction logging). No new engine,
   no new `.claude/`, no new dashboard UI.
3. **Extracted runner seam.** Lift the project-01 runner into
   `curriculum/_shared/benchmarks/runner.py` exposing
   `run_benchmark(project_dir, lang, scenario, run_num) -> dict`, reading a
   per-project `benchmark.yaml` (image name, container port, host-port override,
   scenario list). `analyzer.py` is reused unchanged. The extraction is validated
   against project 01's known-good output as a regression baseline.
4. **Report = aggregation/view.** The arena writes
   `curriculum/NN/docs/arena_report.md` linking the canonical per-language docs;
   raw evidence (`aggregated.json`, scenario results) stays where it is.

## Alternatives Considered

### Alternative 1: Generic multi-language benchmark framework

- **Description**: Harness autodiscovery, container matrices, per-language config
  schemas so any project/language runs out of the box.
- **Pros**: Scales to all 18 projects cleanly.
- **Cons**: Large net-new surface; re-creates the over-engineering that demoted
  the prior `engines/polyglotEvolutionArena/`.
- **Why rejected**: V1 needs only the `run_benchmark(project_dir, lang, scenario,
  run_num)` seam + a per-project `benchmark.yaml`, proven on a few projects.

### Alternative 2: Standalone arena engine / script outside the engine

- **Description**: A separate runner script or engine that calls docker/k6
  directly, bypassing the command/agent framework.
- **Pros**: Fewer moving parts to learn.
- **Cons**: Loses producer ≠ verifier routing, the gate/state conventions, and
  dashboard integration the engine already provides; duplicates orchestration.
- **Why rejected**: The command/agent framework already encodes the guarantees the
  arena needs.

## Consequences

### Positive

- Smallest net-new surface: one command, two agents, one Python module, one
  extracted runner, one metadata file per project.
- `analyzer.py` and its gates are reused verbatim — statistical trust for free.
- Per-project `benchmark.yaml` keeps the dev-machine port-conflict override
  (calibration knob) instead of hardcoding it.

### Negative

- Projects 02 and 05 have no `benchmarks/` harness today; Phase 2 must author
  `benchmark.yaml` + k6 scenarios + bench-target Dockerfiles for them.
- The runner extraction touches working project-01 code; the regression baseline
  is mandatory, not optional.

### Risks

- **Runner extraction breaks project 01** → mitigated by diffing extracted-runner
  output against the committed project-01 `aggregated.json` before adoption.
- **Scenario portability** → scenarios may not transfer verbatim across problem
  shapes; `benchmark.yaml` declares each project's own scenarios.

## Implementation Notes

- `benchmark.yaml` (per project) fields: `images: {go,rust,node: name}`,
  `ports: {go,rust,node: container}`, `host_ports: {…}` (optional override),
  `scenarios: [baseline, stress, …]`.
- `analyzer.py` input contract is unchanged: `{scenario: {lang: [sample,…]}}`.
