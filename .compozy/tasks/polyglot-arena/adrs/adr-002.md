# ADR-002: V1 experience — reveal-time per-metric prediction gate, CLI-triggered

## Status

Accepted

## Date

2026-06-25

## Context

ADR-001 set the V1 scope: a narrative-led projection over existing substrate,
gated by a learner prediction, covering 2–3 projects. The PRD phase pinned the
remaining product decisions:

- **Prediction shape:** per-metric (latency / memory / throughput) winners, not a
  single overall winner — richer calibration data that seeds the V2 meter.
- **V1 projects:** 01 rate_limiter (verified baseline) + 02 key_value_store
  (memory/data-structure) + 05 websocket_chat (concurrency/IO) — three
  contrasting "winner" shapes.
- **Gate strictness:** hard gate — the report stays fully locked until all three
  predictions are logged; no peeking, no anchoring.
- **Make-or-break success bar:** trustworthy lessons — every comparison fair and
  every narrative claim verifier-confirmed; zero wrong lessons shipped.

What remained was the *shape of the V1 experience*: where the learner predicts,
how the report reveals, and how much new surface to build.

## Decision

Ship a **CLI-triggered, reveal-time gate** experience:

1. The learner runs `/devschool-arena <project>`.
2. Agents generate the three implementations, run the fairness audit and
   benchmarks, produce the code study, and draft the narrative; PROMĘTOR verifies
   every narrative claim against the numbers before anything is shown.
3. The learner is prompted to commit **three per-metric predictions**
   (latency/memory/throughput winners). The report is **locked** until all three
   are logged.
4. On commit, the full report reveals as `curriculum/NN/docs/arena_report.md`,
   and the prediction + outcome flow through the existing
   `learner/` → substrate → codexDojo dashboard pipeline.

No new engine, no new dashboard UI. The arena is a command + thin module that
orchestrates existing agents and aggregates existing per-language docs.

## Alternatives Considered

### Alternative 1: Predict-ahead, two-phase

- **Description**: The learner sees the challenge spec and predicts *first* (from
  the problem alone, before agents finish), predictions are logged, the arena
  runs, and the reveal comes later.
- **Pros**: Pedagogically purest anti-anchoring — the prediction is made from
  problem understanding, not from knowing the run is complete.
- **Cons**: Requires a stateful two-step predict→run→reveal flow with persistence
  between steps; more orchestration; a slower feedback loop (predict now, learn
  later).
- **Why rejected**: The hard gate already prevents anchoring at reveal time (the
  learner predicts with results hidden), capturing most of the benefit for far
  less new surface. The two-phase flow is a V2 refinement, not a V1 prerequisite.

### Alternative 2: Dashboard-first interactive

- **Description**: The entire loop lives in the codexDojo SPA — browse projects,
  enter arena, predict in-UI, reveal with an interactive scoreboard and diff
  viewer.
- **Pros**: Richest UX; most discoverable; best home for the V2 calibration meter.
- **Cons**: Significant net-new frontend (scoreboard widget, diff viewer,
  prediction form); directly conflicts with the V1 non-goal "no new dashboard UI
  work."
- **Why rejected**: Breaks the "projection, not a build-out" discipline that keeps
  V1 shippable. The dashboard already renders evidence; V1 reuses it as-is. The
  interactive surface is a clear V2/V3 candidate once the pipeline is proven.

## Consequences

### Positive

- Smallest possible net-new surface: narrative generator + per-metric prediction
  log + the de-hardcoded runner seam. Everything else is reused.
- Hard gate + reveal-time prediction preserves learning-gate integrity with a
  single, fast interaction per run.
- Per-metric predictions produce the calibration corpus the V2 meter needs from
  day one.

### Negative

- The learner predicts knowing the run has completed (results hidden) rather than
  from the bare problem — a slightly weaker form of anti-anchoring than the
  two-phase flow.
- Reveal happens in Markdown + the existing dashboard, not an interactive viewer;
  the side-by-side study is static in V1.

### Risks

- **Prediction friction discourages runs** → mitigated by keeping it to three
  one-tap per-metric picks plus an optional one-line reason.
- **Reveal-time prediction still allows soft anchoring** (the learner knows the
  run is "done") → accepted for V1; the two-phase flow is the V2 upgrade path if
  this proves to matter.

## Implementation Notes

- V1 projects: 01, 02, 05. Project 01 doubles as the runner regression baseline.
- The prediction log lives under `learner/` (auditable Markdown/NDJSON), keyed to
  the project + metric, recording predicted winner, actual winner, and the gap.
